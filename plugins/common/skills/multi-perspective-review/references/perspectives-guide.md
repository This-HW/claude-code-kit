# Perspectives Guide: 10개 관점 상세

> 각 관점의 역할, 책임, 질문 항목

---

## 1. Requirements (기획자)

**역할:** 요구사항 명확화 및 모호함 제거

**책임:**

- P0 모호함 식별 및 질문
- 엣지 케이스 점검
- 비기능 요구사항 확인

**관련 에이전트:** clarify-requirements

**질문 항목:**

```
□ 사용자/대상이 명확한가?
□ 조건이 구체적인가? (언제, 어떤 경우)
□ 결과가 명확한가? (어떻게 되어야)
□ 예외 상황이 정의되었는가?
□ 수치/기준이 명시되었는가?
□ 엣지 케이스 처리가 있는가?
```

**출력 예시:**

```
🔴 P0 모호함:
1. "사용자"가 누구? (회원? 비회원?)
2. 적립률은? (명시 없음)
3. 포인트 사용 제한은?

⚠️ 엣지 케이스:
- 환불 시 포인트 처리?
- 부분 취소 시?
- 포인트 만료 시?
```

## 2. Technical (개발자)

**역할:** 기술적 실현가능성 및 개발 계획 검토

**책임:**

- 기술 스택 적합성
- 예상 개발 기간
- 기존 시스템 충돌 확인

**관련 에이전트:** plan-implementation

**질문 항목:**

```
□ 현재 기술 스택으로 구현 가능한가?
□ 새 의존성이 필요한가?
□ 기존 시스템과 충돌은 없는가?
□ 예상 개발 기간은?
□ 기술 부채가 발생하는가?
□ 테스트 전략은?
```

**출력 예시:**

```
⚠️ 기술적 이슈:
1. 새 테이블 필요 (points, point_transactions)
2. 트랜잭션 무결성 중요 (SERIALIZABLE)
3. 예상 개발: 2-3주

✅ 기존 시스템 충돌 없음
✅ 현재 스택으로 구현 가능
```

## 3. Security (보안 전문가)

**역할:** 보안 취약점 및 리스크 식별

**책임:**

- 인증/권한 체계 검토
- 민감 데이터 처리 확인
- 공격 벡터 분석

**관련 에이전트:** security-scan

**질문 항목:**

```
□ 인증이 필요한가?
□ 권한 체계가 적절한가?
□ 민감 데이터 암호화는?
□ 공격 가능성은? (SQL Injection, XSS 등)
□ 감사 로그가 있는가?
□ Rate Limiting이 필요한가?
```

**출력 예시:**

```
🔴 Critical:
1. 포인트 조작 방지 없음
   → 트랜잭션 격리 필요
2. 중복 적립 방지 로직 없음
   → UUID + 타임스탬프 기반 멱등성
3. 감사 로그 필수
   → who, when, amount 기록

🟡 Important:
- Rate Limiting (IP당 1분 10회)
```

## 4. UX/Flow (UX 디자이너)

**역할:** 사용자 경험 및 흐름 검토

**책임:**

- 사용자 여정 설계
- 상호작용 패턴 확인
- 접근성 점검

**관련 에이전트:** design-user-journey

**질문 항목:**

```
□ 사용자 여정이 명확한가?
□ 진입점이 쉽게 찾을 수 있는가?
□ 피드백이 즉시 제공되는가?
□ 에러 상황에서 어떻게 안내하는가?
□ 모바일/데스크톱 모두 고려되었는가?
□ 접근성(A11y)이 보장되는가?
```

**출력 예시:**

```
✅ 주요 흐름:
1. 구매 완료 → 포인트 적립 알림
2. 결제 → 포인트 사용 선택 → 할인 적용

⚠️ 개선 필요:
- 포인트 잔액 표시 위치 불명확
- 적립 알림 UX 정의 없음
- 사용 가능 포인트 계산 표시 필요
```

## 5. Business Logic (비즈니스 분석가)

**역할:** 비즈니스 규칙 및 정책 검토

**책임:**

- 비즈니스 규칙 명세
- 도메인 로직 정의
- 정책 일관성 확인

**관련 에이전트:** define-business-logic

**질문 항목:**

```
□ 비즈니스 규칙이 명확한가?
□ 도메인 로직이 정의되었는가?
□ 정책이 일관성 있는가?
□ 예외 규칙이 있는가?
□ 규칙 변경 시나리오는?
□ 법적/규제 요구사항은?
```

**출력 예시:**

```
📋 비즈니스 규칙:
1. 적립률: 구매액의 5%
2. 최소 사용: 1,000P
3. 최대 사용: 결제액의 50%
4. 유효기간: 적립일로부터 1년
5. 만료 전 30일 알림

⚠️ 정의 필요:
- 환불 시 포인트 처리 규칙
- 부분 취소 시 차감 기준
```

## 6. Dependencies (아키텍트)

**역할:** 시스템 간 의존성 및 연동 검토

**책임:**

- 외부 시스템 연동
- 라이브러리 의존성
- API 호환성

**관련 에이전트:** analyze-dependencies

**질문 항목:**

```
□ 외부 시스템 연동이 필요한가?
□ 새 라이브러리가 필요한가?
□ API 버전 호환성은?
□ 서비스 간 의존성은?
□ 장애 전파 가능성은?
□ 서킷 브레이커가 필요한가?
```

**출력 예시:**

```
🔗 의존성:
1. Users Service (포인트 잔액)
2. Orders Service (적립 트리거)
3. Payments Service (포인트 사용)

⚠️ 주의:
- Orders → Payments 강결합
- Payments 장애 시 포인트 사용 불가
→ 제안: 이벤트 기반 느슨한 결합
```

## 7. Code Quality (코드 리뷰어)

**역할:** 코드 품질 및 유지보수성 검토

**책임:**

- 코드 가독성
- 테스트 가능성
- 유지보수성

**관련 에이전트:** review-code

**질문 항목:**

```
□ 코드가 읽기 쉬운가?
□ 단일 책임 원칙을 따르는가?
□ 테스트 가능한 구조인가?
□ 중복 코드가 없는가?
□ 네이밍이 명확한가?
□ 에러 처리가 적절한가?
```

**출력 예시:**

```
✅ 긍정:
- 트랜잭션 로직 명확히 분리
- 비즈니스 규칙 상수화

⚠️ 개선:
- 포인트 계산 로직 유틸 분리 권장
- 에러 타입 정의 필요
- 테스트 커버리지 80% 이상 권장
```

## 8. Metrics (데이터 엔지니어)

**역할:** 성능 지표 및 모니터링 검토

**책임:**

- 성능 목표 정의
- 모니터링 설계
- SLA 확인

**관련 에이전트:** define-metrics

**질문 항목:**

```
□ 성능 목표가 정의되었는가?
□ 모니터링 항목은?
□ SLA가 명시되었는가?
□ 성능 병목이 예상되는가?
□ 로깅 전략은?
□ 알림 기준은?
```

**출력 예시:**

```
📊 성능 목표:
- 포인트 적립: < 100ms
- 포인트 조회: < 50ms
- 결제 시 포인트 사용: < 200ms

📈 모니터링:
- 포인트 적립/사용 건수 (일/주/월)
- 평균 적립 포인트
- 포인트 사용률
- 이상 패턴 (대량 적립/사용)

⚠️ 알림:
- 응답시간 > 500ms
- 에러율 > 1%
- 1분 내 100건 이상 적립/사용
```

## 9. Data/Schema (DB 설계자)

**역할:** 데이터 모델 및 스키마 검토

**책임:**

- 스키마 설계
- 데이터 정합성
- 마이그레이션 계획

**관련 에이전트:** design-database

**질문 항목:**

```
□ 스키마가 정규화되었는가?
□ 인덱스 설계는 적절한가?
□ 데이터 정합성 제약은?
□ 마이그레이션 전략은?
□ 백업/복구 계획은?
□ 데이터 보관 정책은?
```

**출력 예시:**

```
📊 스키마:

points:
  - user_id (FK)
  - balance (INT)
  - created_at, updated_at

point_transactions:
  - id (PK)
  - user_id (FK)
  - type (EARN/USE/EXPIRE)
  - amount
  - order_id (nullable)
  - created_at

✅ 인덱스:
- points: (user_id) UNIQUE
- point_transactions: (user_id, created_at)

⚠️ 주의:
- 트랜잭션 로그 증가 → 파티셔닝 고려 (월별)
- 만료 포인트 배치 처리 (매일 0시)
```

## 관점 간 협업 패턴

### 협업 1: Requirements ↔ Technical

```
Requirements: "사용자 정의 불명확"
Technical: "회원/비회원 모두 지원하려면 인증 분기 필요, 개발 +1주"
→ 합의: 회원만 Phase 1, 비회원 Phase 2
```

### 협업 2: Security ↔ UX

```
Security: "2FA 필수"
UX: "사용자 이탈 우려"
→ 합의: 리스크 기반 인증 (금액별)
```

### 협업 3: Technical ↔ Data/Schema

```
Technical: "트랜잭션 무결성 필요"
Data/Schema: "SERIALIZABLE 격리 레벨, 인덱스 (user_id) UNIQUE"
→ 합의: 설계 확정
```

## 10. Devil's Advocate (전략 비평가)

**역할:** 설계/아키텍처 수준 적대적 공격

**책임:**

- 실패 시나리오 발굴
- 리스크 정량화
- 완화 방안 제시

**관련 에이전트:** devils-advocate

**질문 항목:**

```
□ 이 설계가 실패할 수 있는 이유는?
□ 규모가 10배 커지면 어떻게 되는가?
□ 외부 의존성이 망가지면?
□ 6개월 후 유지보수 가능한가?
□ 개발/운영 비용은 감당 가능한가?
```

**출력 형식:**

```
[HIGH] FAIL-001: [시나리오 제목]
- 페르소나: [확장성 파괴자 | 의존성 파괴자 | 미래의 유지보수자 | 비용 감시자]
- 트리거: [조건]
- 실패 과정: [단계별]
- 영향: [비즈니스 영향]
- 확률: High/Medium/Low
- 완화 방안: [즉시/중기/장기]
```

**사용 시점:**

```
✅ Large 규모 (자동 포함)
✅ Medium 규모 + 아키텍처 변경 (선택적)
✅ 전략적 리스크가 높은 변경
```
