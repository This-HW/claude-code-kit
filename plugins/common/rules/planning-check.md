# 기획 확인 규칙

> 애매하거나 불확실한 부분이 있으면 반드시 기획을 재확인합니다.
> 추측으로 구현하지 않습니다.

## 핵심 원칙

**"확실하지 않으면, 멈추고 확인한다"**

```
❌ "아마 이렇게 동작해야 할 것 같아서..." / "보통 이런 식으로 하니까..."
✅ "기획서 확인 결과 [X]로 동작해야 합니다" / "명세에 따르면 [Y] 조건에서 [Z] 수행"
```

## 기획 확인이 필요한 상황

1. **요구사항 불명확**: "~할 것 같다", "보통은 ~" → 즉시 기획 문서 확인
2. **엣지 케이스**: 빈 값/오류/권한 없음 시 동작 → 기획서 확인, 없으면 사용자에게 질문
3. **다중 해석**: "사용자 정보 표시" (어떤 정보? 형식?) → 명확한 정의 확인
4. **비즈니스 로직**: 할인 계산, 권한 체계, 상태 전이 → 반드시 기획서/명세 기반 구현

## 기획 확인 워크플로우

1. **불확실성 감지**: "이게 맞나?", "기획에 이 부분이 있었나?" → 즉시 멈춤
2. **기획 문서 검색**: `docs/planning/`, 외부 도구(Notion MCP, Figma MCP), GitHub Issues
3. **정보 부재 시**: AskUserQuestion으로 사용자에게 질문 (옵션 A/B 제시)
4. **결정 기록**: 코드 주석에 결정 내용과 근거 기록

## 확인 체크리스트

**구현 전:** 요구사항 문서 존재 / 성공·실패·로딩·빈 값 상태 정의 / 엣지 케이스 명시 / UI/UX 스펙

**구현 중:** 추측으로 구현하지 않음 / 기획과 다르게 구현하지 않음 / 기획에 없는 기능 추가하지 않음

**구현 후:** 구현 결과가 기획과 일치 / 모든 케이스가 기획대로 동작

## 질문 형식 (기획 부재 시)

```
[기능명]에 대해 확인이 필요합니다.
상황: [현재 구현하려는 것]
불명확한 점: [구체적인 질문]
옵션: A) [해석 1]  B) [해석 2]
기획 문서에서 관련 내용을 찾지 못했습니다. 어떤 방향으로 진행할까요?
```
