# Planning 협업 프로토콜

> Planning과 Dev 간의 협업 규칙을 정의합니다. 모든 에이전트는 이 프로토콜을 준수해야 합니다.

## 핵심 원칙

### 1. 추측하지 않는다

```
❌ "아마 사용자는 ~를 원할 것이다" / "~일 것 같다"
✅ "사용자가 명시한 요구사항은 ~이다" / "기획 문서에 따르면 ~이다"
```

### 2. 모호함을 분류한다 (P0~P3)

| 등급 | 기준 | 예시 | 처리 |
|------|------|------|------|
| **P0** | 데이터 무결성, 보안, 금융, 핵심 비즈니스 | 결제 금액 계산, 권한 체계, 데이터 삭제 정책 | **즉시 중단 + 질문** |
| **P1** | UX 분기, 비즈니스 디테일 | 에러 메시지 문구, 정렬 순서 | 기본값 + 나중에 확인 |
| **P2** | UI 디테일, 엣지케이스 | 버튼 색상, 극단적 케이스 | TODO 기록 |
| **P3** | 기술 선택 | 라이브러리, 패턴 | 자율 판단 |

### 3. Dev → Planning 역위임 프로토콜

구현 중 기획 모호함 발견 시:

| 유형 | 처리 |
|------|------|
| P0_AMBIGUITY | Planning으로 돌아가 AskUserQuestion 사용 |
| MISSING_SPEC | 해당 명세 추가 (여정/규칙) |
| INFEASIBLE | 대안 검토 → 사용자에게 보고 |

## 작업 규모 판단 기준

| 기준 | Small | Medium | Large |
|------|-------|--------|-------|
| **영향 범위** | 1개 모듈 내 | 2-3개 모듈 | 4개 이상 |
| **예상 파일 수** | 1-3개 | 4-10개 | 10개+ |
| **데이터 변경** | 없음 | 기존 구조 확장 | 새 구조 추가 |
| **비즈니스 규칙** | 기존 내 | 경미한 추가 | 핵심 변경 |
| **예상 공수** | ~10h | 20-50h | 50h+ |

**규모별 필수 단계:** Small: 요구사항만 / Medium: +사용자 여정 / Large: +비즈니스 로직

## Planning 완료 조건

**모든 규모:** P0 모호함 = 0 / 핵심 요구사항 정의 / 영향 범위 식별 / 리스크 분석

**Medium+:** 사용자 여정 / 주요 상태 전이 / 에러 처리 전략

**Large+:** 비즈니스 규칙 / 규칙 간 관계 / 예외 처리

## P0 질문 프로토콜

P0 발견 시: 즉시 중단 → AskUserQuestion 사용 → 답변 기록 → Planning 재진행

**질문 형식:**

```
**맥락**: [상황]  **질문**: [구체적 질문]
**옵션**: 1. [A]  2. [B]
```

## 정보 전달 형식

**Planning → Dev 인수인계 항목:** 요구사항 요약 / 확인된 사항 / 사용자 여정(해당시) / 비즈니스 규칙(해당시) / 미결정 P1/P2 / 구현 권장사항

**Dev → Planning 역위임 항목:** 유형(P0_AMBIGUITY|MISSING_SPEC|INFEASIBLE) / 발생 컨텍스트 / 발견한 문제 / 제안 옵션 / 필요한 결정

## 모호함 탐지 신호

**위험:** "~할 것 같다", "아마 ~", "보통은 ~", "임시로 ~"

**안전:** "기획에 따르면", "사용자가 요청한", "확인 결과"

## 체크리스트

**Planning 시작 전:** 요청 명확 / 규모 판단 / 필요 단계 식별

**Planning 진행 중:** P0 즉시 질문 / 각 단계 결과 기록 / 다음 단계 정보 전달

**Planning 완료 시:** P0 모두 해결 / 인수인계 문서 / P1/P2 기록 / Dev 시작 준비
