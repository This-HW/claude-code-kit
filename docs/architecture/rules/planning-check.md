# 기획 확인 규칙

> 이 문서는 구현 전 기획을 확인해야 하는 상황과 확인 방법을 설명합니다.

---

## 1. 핵심 원칙

```
NEVER implement based on assumption.
ALWAYS stop and verify specs first.
```

추측으로 구현한 코드는 나중에 전부 다시 작성하게 됩니다. 5분 확인이 5시간 재작업을 막습니다.

---

## 2. 기획 확인이 필요한 4가지 상황

### 상황 1: 요구사항 불명확

**판단 기준:** "~할 것 같다", "보통은 ~" 같은 표현이 머릿속에 떠오를 때

**실전 예시:**

```
# 잘못된 접근
"비밀번호 찾기는 보통 이메일로 링크 보내는 방식이니까 그냥 구현하자"
→ 실제 기획: SMS OTP 방식 사용

# 올바른 접근
"비밀번호 찾기 방식이 명세에 없다 → 기획서 확인 또는 사용자 질문"
```

---

### 상황 2: 엣지 케이스

**판단 기준:** 정상 케이스만 정의되어 있고, 예외 상황(빈 값, 오류, 권한 없음)이 명세에 없을 때

**실전 예시:**

```
명세: "사용자 목록을 표시한다"

엣지 케이스 질문:
- 사용자가 0명일 때? (빈 상태 메시지)
- 로딩 중? (스켈레톤 UI vs 스피너)
- API 오류? (재시도 버튼 vs 에러 페이지)
- 권한 없는 사용자가 접근? (403 vs 로그인 리디렉션)
```

---

### 상황 3: 다중 해석 가능

**판단 기준:** 같은 요구사항을 두 가지 이상으로 해석할 수 있을 때

**실전 예시:**

```
요구사항: "최근 주문을 표시한다"

해석 A: 최근 30일 주문
해석 B: 최근 10개 주문
해석 C: 사용자의 전체 주문 (페이지네이션)

→ 반드시 확인 필요
```

---

### 상황 4: 비즈니스 로직

**판단 기준:** 할인 계산, 권한 체계, 상태 전이 등 비즈니스 규칙이 관련될 때

**실전 예시:**

```
# 반드시 기획서/명세 기반으로 구현해야 하는 케이스
- 쿠폰 중복 적용 가능 여부
- 관리자 vs 일반 사용자 권한 차이
- 주문 취소 가능한 상태 범위 (결제 완료 후 취소 가능?)
- 포인트 적립 계산식
```

## 4. 올바른 vs 잘못된 접근 예시

### 잘못된 접근 — 추측 기반 구현

```typescript
// ❌ 잘못된 예: 명세 없이 추측으로 구현
function applyDiscount(price: number, coupon: Coupon): number {
  // "보통은 한 번에 하나만 쓰겠지" 라고 추측
  return price - coupon.discountAmount;
}

// 실제 요구사항: 쿠폰은 최대 2개 중복 적용 가능, 단 총 할인율 50% 초과 불가
// → 전체 재작업 필요
```

### 올바른 접근 — 명세 확인 후 구현

```typescript
// ✅ 올바른 예: 명세 확인 후 구현
// 기획서 v2.3 참조: 쿠폰 최대 2개 중복 적용, 총 할인율 50% 상한선
function applyDiscounts(price: number, coupons: Coupon[]): number {
  if (coupons.length > 2) {
    throw new ValidationError("COUPON_LIMIT", "최대 2개까지 적용 가능합니다");
  }

  const totalDiscount = coupons.reduce((sum, c) => sum + c.discountRate, 0);
  const cappedDiscount = Math.min(totalDiscount, 0.5); // 50% 상한선

  return Math.floor(price * (1 - cappedDiscount));
}
```

## 5. 기획 부재 시 질문 형식 템플릿

기획 문서를 찾지 못했을 때 사용자에게 질문하는 형식입니다.

```
[기능명]에 대해 확인이 필요합니다.

상황: [현재 구현하려는 것과 왜 결정이 필요한지]

불명확한 점: [구체적인 질문]

옵션:
  A) [해석 1] — [이 방식의 특징/영향]
  B) [해석 2] — [이 방식의 특징/영향]
```

**실전 예시 1 — 엣지 케이스:**

```
장바구니 기능에 대해 확인이 필요합니다.

상황: 장바구니 아이템 수량 수정 기능을 구현 중입니다.

불명확한 점: 수량을 0으로 설정했을 때의 동작

옵션:
  A) 수량 0 입력 시 아이템 자동 삭제 — 일반적인 쇼핑몰 방식
  B) 수량 0 유지 (0개 상태 표시) + 별도 삭제 버튼 필요
```

**실전 예시 2 — 비즈니스 로직:**

```
회원 등급 혜택에 대해 확인이 필요합니다.

상황: 회원 등급별 할인 적용 로직을 구현 중입니다.

불명확한 점: 쿠폰 할인과 회원 등급 할인의 중복 적용 여부

옵션:
  A) 중복 적용 가능 — 각각 독립적으로 계산 후 합산
  B) 중복 불가 — 더 큰 할인만 적용
  C) 순서 적용 — 회원 할인 먼저, 그 후 쿠폰 적용
```

## 6. 결정 근거 코드 주석 형식

기획 확인 후 구현할 때 결정 근거를 코드에 남깁니다.

```typescript
// [기획 확인] 2024-01-15 사용자 확인
// 질문: 쿠폰 중복 적용 가능 여부
// 결정: 최대 2개 중복 적용 가능, 총 할인율 50% 상한
// 근거: 기획서 v2.3 또는 담당자 이메일 확인
const MAX_COUPONS = 2;
const MAX_DISCOUNT_RATE = 0.5;
```

## 7. 구현 전·중·후 체크리스트

### 구현 전

- [ ] 요구사항 문서가 존재하는가?
- [ ] 성공 케이스가 정의되어 있는가?
- [ ] 실패/오류 케이스가 정의되어 있는가?
- [ ] 로딩/빈 값 상태가 정의되어 있는가?
- [ ] 엣지 케이스가 명시되어 있는가?

### 구현 중

- [ ] 추측으로 구현하고 있지는 않은가?
- [ ] 명세에 없는 기능을 추가하고 있지는 않은가?
- [ ] 명세와 다른 방향으로 구현하고 있지는 않은가?

### 구현 후

- [ ] 구현 결과가 기획과 일치하는가?
- [ ] 모든 케이스(성공/실패/로딩/빈값)가 기획대로 동작하는가?
- [ ] 결정 사항이 코드 주석에 기록되어 있는가?
