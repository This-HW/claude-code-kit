# Planning 협업 프로토콜

> 이 문서는 Planning 단계에서의 모호함 처리, 작업 규모 판단, Dev와의 인수인계 방식을 설명합니다.

---

## 1. 핵심 원칙

```
NEVER implement based on assumption.
ALWAYS verify against specs or ask the user.
```

불명확한 상태로 구현을 시작하면 나중에 전체를 다시 작성해야 합니다. Planning에 시간을 쓰는 것이 Dev 재작업보다 항상 효율적입니다.

---

## 2. 모호함 등급 (P0 ~ P3)

| 등급   | 범위                                  | 처리 방법                   | 실전 케이스                                                |
| ------ | ------------------------------------- | --------------------------- | ---------------------------------------------------------- |
| **P0** | 데이터 무결성·보안·금융·핵심 비즈니스 | **즉시 중단 + 사용자 질문** | 결제 환불 정책, 데이터 삭제 방식, 권한 체계, 비밀번호 정책 |
| **P1** | UX 분기·비즈니스 디테일               | 기본값 적용 후 나중에 확인  | 오류 시 토스트 vs 모달, 페이지네이션 기본 갯수             |
| **P2** | UI 디테일·엣지케이스                  | TODO 주석으로 기록          | 빈 목록 메시지 문구, 날짜 표시 형식                        |
| **P3** | 기술 선택 (라이브러리·패턴)           | 자율 판단, 결정 이유 기록   | ORM 선택, 상태관리 라이브러리, 폴더 구조                   |

### P0 질문 형식

P0가 발견되면 즉시 멈추고 아래 형식으로 사용자에게 질문합니다:

```
**맥락**: [현재 구현하려는 것과 왜 결정이 필요한지]
**질문**: [구체적인 결정 사항]
**옵션**:
  1. [선택지 A] — [A를 선택했을 때의 영향]
  2. [선택지 B] — [B를 선택했을 때의 영향]
```

**실전 예시:**

```
**맥락**: 사용자 계정 삭제 기능을 구현하려 합니다. 삭제 방식에 따라
  데이터베이스 스키마와 쿼리 로직이 크게 달라집니다.

**질문**: 사용자 계정 삭제 시 어떤 방식을 사용하나요?

**옵션**:
  1. 소프트 삭제 (deletedAt 컬럼) — 데이터 보존, 복구 가능, 쿼리에
     WHERE deletedAt IS NULL 필요
  2. 하드 삭제 (물리적 삭제) — 완전 삭제, GDPR 준수 용이, 복구 불가
```

## 3. Dev → Planning 역위임 프로토콜

구현 중에 기획 모호함을 발견하면 Planning으로 돌아가야 합니다. 아래 흐름을 따릅니다.

```
┌──────────────────────────────────────────────────────────┐
│  Dev 에이전트 (implement-code 등)                        │
│  구현 중 기획 모호함 발견                                │
└──────────────┬───────────────────────────────────────────┘
               │
               ▼ 유형 분류
        ┌──────────────────────────────┐
        │  P0_AMBIGUITY                │ → AskUserQuestion 즉시 실행
        │  핵심 비즈니스 규칙 불명확   │
        └──────────────────────────────┘
        ┌──────────────────────────────┐
        │  MISSING_SPEC                │ → Planning 에이전트 재호출
        │  명세 자체가 존재하지 않음   │   (여정·규칙 추가)
        └──────────────────────────────┘
        ┌──────────────────────────────┐
        │  INFEASIBLE                  │ → 대안 검토 후 사용자 보고
        │  기술적으로 불가능하거나     │
        │  비용이 너무 큼              │
        └──────────────────────────────┘
```

**Dev → Planning 역위임 신호 예시:**

```
---DELEGATION_SIGNAL---
TYPE: NEED_USER_INPUT
TARGET: (없음)
REASON: P0_AMBIGUITY — 장바구니 수량 0 설정 시 처리 방법 미정의
CONTEXT:
  발생 위치: CartService.updateQuantity()
  발견한 문제: 수량을 0으로 설정했을 때 아이템을 삭제해야 하는지,
               0개 상태로 유지해야 하는지 명세 없음
  제안 옵션:
    1. 수량 0 = 아이템 삭제 (일반적인 전자상거래 방식)
    2. 수량 0 유지 + 별도 삭제 버튼 필요
  필요한 결정: 위 두 가지 중 선택
---END_SIGNAL---
```

## 4. 작업 규모 판단 기준

Planning 산출물의 깊이는 작업 규모에 따라 달라집니다.

| 규모       | 기준                               | 필요 산출물                     | 실전 예시                                  |
| ---------- | ---------------------------------- | ------------------------------- | ------------------------------------------ |
| **Small**  | 모듈 1개, 파일 1~3개, ~10시간      | 요구사항 목록만                 | 단일 API 엔드포인트 추가, 버튼 스타일 변경 |
| **Medium** | 모듈 2~3개, 파일 4~10개, 20~50시간 | 요구사항 + 사용자 여정          | 사용자 인증 시스템, 게시판 CRUD            |
| **Large**  | 모듈 4개+, 파일 10개+, 50시간+     | 요구사항 + 여정 + 비즈니스 규칙 | 결제 시스템, 멀티테넌트 권한 체계          |

## 5. Planning 완료 조건

Dev에 인수인계하기 전 아래 조건을 모두 충족해야 합니다.

### 공통 (모든 규모)

- [ ] P0 모호함 0개
- [ ] 핵심 요구사항 정의 완료
- [ ] 영향 범위 식별 (어떤 모듈·파일이 변경되는가)
- [ ] 리스크 분석 완료 (무엇이 잘못될 수 있는가)

### Medium 이상 추가 조건

- [ ] 사용자 여정 (User Journey) 작성
- [ ] 주요 상태 전이 정의 (예: 주문 → 결제 → 배송 → 완료)
- [ ] 에러 처리 전략 포함 (각 실패 케이스별 처리 방법)

### Large 추가 조건

- [ ] 비즈니스 규칙 명세 (예: 할인 중복 적용 규칙, 환불 계산 공식)
- [ ] 규칙 간 관계 정의 (규칙 A와 B가 충돌할 때)
- [ ] 예외 처리 정의 (정책의 예외 케이스)

## 6. Planning → Dev 인수인계 문서 템플릿

Dev 에이전트에게 전달하는 컨텍스트 형식입니다.

```markdown
## 요구사항 요약

[1~3문장으로 무엇을 만들어야 하는지]

## 확인된 사항 (P0 해결 결과)

- [결정 사항 1]: [사용자가 선택한 옵션과 이유]
- [결정 사항 2]: [사용자가 선택한 옵션과 이유]

## 사용자 여정 (Medium+ 해당)

1. 사용자가 [액션]을 하면
2. 시스템이 [반응]을 하고
3. 사용자는 [결과]를 본다

## 비즈니스 규칙 (Large 해당)

- [규칙 1]: [구체적인 계산/검증 로직]
- [규칙 2]: [구체적인 계산/검증 로직]

## 미결정 P1/P2 사항

- P1: [항목] — 기본값: [기본값으로 진행]
- P2: [항목] — TODO로 처리

## 구현 권장사항

- [기술적 고려사항 1]
- [기술적 고려사항 2]
```

## 7. 모호한 언어 금지

Planning 문서와 에이전트 출력에서 아래 표현은 사용 금지입니다.

| 금지 표현     | 이유                        | 대체 표현                  |
| ------------- | --------------------------- | -------------------------- |
| "~할 것 같다" | 추측 기반 구현 유발         | "기획에 따르면 ~"          |
| "아마 ~"      | 불확실성 숨김               | "확인 결과 ~"              |
| "보통은 ~"    | 일반론으로 특수 케이스 무시 | "이 시스템에서는 ~"        |
| "임시로 ~"    | 기술 부채 암묵적 허용       | "TODO: [이유]로 추후 변경" |
| "대충 ~"      | 정밀도 저하                 | 구체적 수치/조건 명시      |
